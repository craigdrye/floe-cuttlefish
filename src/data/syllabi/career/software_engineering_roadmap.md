# Software Engineering Roadmap
**ID**: `softwareEngineeringRoadmap` - **Discipline**: Technology - **Level**: Career

## Course Aim
Help learners understand software-engineering workplace language: architecture review, technical debt, migration, incident follow-up, release risk, observability, testing, and tradeoffs. The course is designed for people entering technical teams who need to follow the conversation and contribute clearly.

## Course Design Notes
Route questions here when they test engineering process, system design, code review, testing, releases, migrations, incidents, technical debt, or architecture communication. Keep the course practical and cross-functional rather than deep language-specific coding.

## Chapter 1: Code, Review, and Quality
- **Key concepts**: pull request, code review, unit test, integration test, regression, refactor, maintainability.
- **Practice questions**: What risk does the change create? What evidence shows it works?
- **Work output**: code-review checklist.

## Chapter 2: Architecture and Tradeoffs
- **Key concepts**: service boundary, API contract, coupling, scalability, latency, consistency, dependency.
- **Practice questions**: What is the tradeoff? What constraint matters most?
- **Work output**: architecture decision record.

## Chapter 3: Releases and Migrations
- **Key concepts**: deploy, rollback, feature flag, migration, compatibility, canary, release note.
- **Practice questions**: What can fail? How can we ship gradually? How do we recover?
- **Work output**: release risk plan.

## Chapter 4: Incidents and Reliability
- **Key concepts**: outage, severity, on-call, monitoring, alert, root cause, remediation, postmortem.
- **Practice questions**: What is user impact? What restores service? What prevents recurrence?
- **Work output**: incident timeline and follow-up plan.

## Chapter 5: Technical Debt and Team Communication
- **Key concepts**: debt, maintenance cost, roadmap tradeoff, ownership, documentation, stakeholder update.
- **Practice questions**: Which debt blocks delivery? Which cleanup is worth prioritizing?
- **Work output**: technical-debt brief with business impact.

## Capstone
Turn a messy engineering planning thread into an implementation plan: scope, architecture decision, tests, release path, rollback, and stakeholder update.
