# Algorithms
**ID**: `algorithms` - **Discipline**: Computing - **Stage**: University Sophomore
**Aligns with**: standard CS2 / algorithms courses (CLRS-based) and technical interview foundations

## Course Aim
This course turns programmers into algorithm designers. The difference is not knowing more tricks; it is the habit of stating a problem precisely, choosing a design paradigm deliberately, arguing that the result is correct, and predicting its cost before running it. By the end, a student can look at an unfamiliar problem and ask the right questions in order: what is the input and its size, what counts as a feasible and an optimal answer, which paradigm fits the structure, and what asymptotic bound the chosen approach buys.

The intellectual core is the marriage of three disciplines that students usually keep separate. Analysis gives a vocabulary (Big-O, Theta, Omega) for talking about growth instead of stopwatch seconds. Proof gives loop invariants, induction, exchange arguments, and cut properties so that "it passed my tests" becomes "it is correct on every input." Engineering keeps the theory honest by forcing students to confront constant factors, cache behavior, adversarial inputs, and the difference between an algorithm that is asymptotically optimal and one that actually finishes on the data in front of them.

The course deliberately resists the flashcard trap that haunts interview prep. A student who has memorized that "Dijkstra is O((V+E) log V)" but cannot explain why a negative edge breaks it, or who can recite the knapsack recurrence but cannot identify the state for a new problem, has not learned algorithms. Every chapter therefore pairs a technique with the judgment about when it applies, the proof that it works, and the failure mode that exposes whether a student truly understands it.

## Course Design Notes
This is a proof-and-implementation algorithms course for students who already know basic programming and the standard data structures (arrays, linked lists, stacks, queues, hash tables, binary search trees, heaps). It is not a data-structures course; route students who need heap internals, tree balancing, or hash-collision strategy elsewhere. Route here when the need is algorithm selection, recurrence analysis, asymptotic comparison, correctness proofs, graph modeling, dynamic-programming state design, reductions, NP-completeness intuition, randomized tradeoffs, approximation guarantees, or technical-interview reasoning that goes beyond pattern matching.

The chapter order is dependency-driven: analysis and correctness first (Chapter 1), because every later claim rests on them; then recurrences and divide-and-conquer (Chapter 2), which supply the analysis machinery for everything recursive; then the two great optimization paradigms, greedy (Chapter 3) and dynamic programming (Chapter 4), taught back to back so students learn to tell them apart; then graphs as a modeling language (Chapter 5) and the weighted-graph algorithms that build on it (Chapter 6); then the theory of intractability (Chapter 7) that explains why some problems resist all of the above; and finally the pragmatic toolkit (Chapter 8) for living with hard problems through randomization, approximation, and measurement. Questions should reward precise problem statements, stated preconditions and invariants, proof sketches over hand-waving, adversarial test design, and written explanations that connect an implementation detail to a theoretical guarantee.

## Chapter 1: Algorithmic Analysis and Correctness
- **Core questions**: What does it mean for one algorithm to be "faster" than another when machines and constants differ? How do we prove an algorithm is correct on every input, not just the ones we tested? What is the difference between an upper bound on a problem and an upper bound on an algorithm?
- **Key concepts**: computational problems and input size; worst-case, best-case, and average-case analysis; Big-O, Big-Theta, Big-Omega, and little-o; the difference between asymptotic tightness (Theta) and a mere upper bound (O); space complexity; loop invariants (initialization, maintenance, termination); proof by induction for correctness; the RAM model and its hidden assumptions.
- **Applied skills**: rank a list of functions by growth rate; analyze nested loops whose inner bound depends on the outer index; write and verify a loop invariant for linear or binary search; instrument code and plot runtime against input size to confirm a predicted curve.
- **Common traps**: saying an algorithm "is O(n^2)" when you mean Theta, since O is only an upper bound and 5n is technically O(n^2); confusing the complexity of a problem with that of one algorithm for it; assuming a passing test suite proves correctness; treating average-case and worst-case as interchangeable; ignoring that "constant time" array access is a model assumption, not a law of physics.

## Chapter 2: Recurrences, Divide and Conquer, and Sorting
- **Core questions**: When you split a problem into pieces, how does the cost of splitting and combining trade off against the cost of the subproblems? Why can no comparison-based sort beat n log n? When is the asymptotically slower algorithm the right engineering choice?
- **Key concepts**: recurrence relations; the substitution method and recursion trees; the Master Theorem and its three cases plus the regularity condition in Case 3; binary search; merge sort; quicksort with deterministic vs. randomized partitioning; quickselect and median-of-medians selection; the Omega(n log n) comparison-sort lower bound via the decision tree; stability and in-place behavior.
- **Applied skills**: solve a given recurrence by two independent methods and check they agree; prove binary search correct with an invariant on the remaining range; trace quicksort's partition on adversarial (already-sorted) input; choose a sort for a constrained workload (nearly-sorted data, bounded integer keys, external memory).
- **Common traps**: applying the Master Theorem when the subproblems are unequal sizes or a, b are not constants (it does not apply); forgetting the Case 3 regularity condition; claiming quicksort is "O(n log n)" without the word "expected" or "average," since its worst case is O(n^2); confusing the linear-expected quickselect with the linear-worst-case median-of-medians; assuming any O(n log n) sort dodges the comparison lower bound (counting/radix sorts escape it only by not comparing).

## Chapter 3: Greedy Algorithms
- **Core questions**: When does taking the locally best choice at every step actually produce a globally optimal answer? How do you prove a greedy rule is optimal rather than merely plausible? When a greedy heuristic is not optimal, can you exhibit the input that breaks it?
- **Key concepts**: the greedy-choice property and optimal substructure; exchange arguments and "greedy stays ahead" proofs; interval scheduling and activity selection (earliest finish time); Huffman coding; fractional knapsack; the matroid framework as the deep reason greedy works; the cut property as a greedy justification (preview of MST).
- **Applied skills**: prove interval scheduling optimal with an exchange argument; build a Huffman tree from a frequency table and read off the code lengths; given a new problem, decide whether it is greedy, dynamic programming, or neither; construct a concrete counterexample for a tempting-but-wrong greedy rule.
- **Common traps**: assuming a problem is greedy because a greedy rule gives a good answer on small examples (e.g., greedy fails on 0/1 knapsack though it solves fractional knapsack); confusing "earliest start" with the correct "earliest finish" rule in interval scheduling; presenting an algorithm trace as a proof of optimality; forgetting that greedy needs both the greedy-choice property and optimal substructure, not just one.

## Chapter 4: Dynamic Programming
- **Core questions**: What makes a problem solvable by remembering answers to subproblems? How do you define a state so that a recurrence over states is both correct and efficient? How do you recover the actual solution, not just its optimal value?
- **Key concepts**: overlapping subproblems vs. optimal substructure; state definition and transition design; memoization (top-down) vs. tabulation (bottom-up); evaluation order and dependency; solution reconstruction via back-pointers; space compression (2D to 1D); canonical problems: longest common subsequence, edit distance, 0/1 knapsack, matrix-chain multiplication, weighted interval scheduling; the connection to shortest paths (Bellman-Ford as DP).
- **Applied skills**: identify the state and transition for an unfamiliar problem; convert a recursive recurrence into a filled table with correct base cases and order; reconstruct an optimal solution from the DP table; compress a two-dimensional table to one dimension when the recurrence only looks back one row.
- **Common traps**: defining a state that does not capture enough information to make the transition correct (under-specified state); choosing an evaluation order that reads a cell before it is computed; reporting the optimal value but being unable to reconstruct the solution; using DP where greedy suffices (wasteful) or greedy where DP is required (wrong); double-counting in the knapsack 0/1 vs. unbounded distinction.

## Chapter 5: Graph Modeling, Search, and Structure
- **Core questions**: How do you turn a real scheduling, dependency, or reachability problem into a graph? When should you store a graph as an adjacency list versus a matrix? What structural facts (cycles, components, bipartiteness, a topological order) can a single traversal reveal?
- **Key concepts**: directed vs. undirected, weighted vs. unweighted graphs; adjacency lists vs. matrices and the sparse/dense tradeoff; BFS and DFS; the DFS edge classification (tree, back, forward, cross) and what a back edge means; connected components; cycle detection in directed and undirected graphs; bipartite testing by 2-coloring; topological sort of a DAG; strongly connected components (preview).
- **Applied skills**: model a prerequisite or dependency problem as a DAG and topologically sort it; trace BFS and DFS, recording discovery/finish times; prove the BFS layer-distance property; detect a cycle by finding a back edge during DFS; pick a representation given V, E, and the dominant query.
- **Common traps**: using a back edge to detect cycles in an undirected graph without excluding the edge to the parent (every undirected tree edge looks like a back edge otherwise); assuming a topological order exists for a graph that has a cycle; choosing an adjacency matrix for a sparse graph and wasting O(V^2) space; conflating "connected" (undirected) with "strongly connected" (directed); thinking DFS finds shortest paths (it does not).

## Chapter 6: Shortest Paths, Minimum Spanning Trees, and Network Problems
- **Core questions**: Why does Dijkstra's greedy expansion fail the moment a negative edge appears, and what does Bellman-Ford do differently? Why is the minimum spanning tree problem solvable greedily when so many graph problems are not? What does union-find buy Kruskal?
- **Key concepts**: single-source shortest paths; Dijkstra's algorithm with a binary heap (O((V+E) log V)) and the Fibonacci-heap bound (O(E + V log V)); Bellman-Ford (O(VE)) and negative-cycle detection; path reconstruction via predecessors; minimum spanning trees; the cut property and the cycle property; Kruskal's algorithm with union-find (path compression and union by rank); Prim's algorithm.
- **Applied skills**: give the concrete graph where Dijkstra returns a wrong distance under negative edges; trace union-find with path compression and report the near-constant amortized cost; prove the MST cut property; choose between a shortest-path and a spanning-tree formulation for a network-design scenario.
- **Common traps**: believing Dijkstra "just needs a tweak" for negative edges (it can settle a vertex too early and never revisit it); confusing a minimum spanning tree with a shortest-path tree (they are generally different); assuming MST is unique when edge weights tie; forgetting Bellman-Ford needs V-1 relaxation rounds and a Vth round only to detect negative cycles; quoting Dijkstra's bound as O(V^2) or O(E log V) without stating the priority-queue implementation.

## Chapter 7: Complexity, Reductions, and NP-Completeness
- **Core questions**: What exactly does it mean to say a problem is "hard"? How does reducing problem A to problem B transfer hardness, and in which direction? What does NP-completeness actually claim about a specific instance you need to solve tomorrow?
- **Key concepts**: decision vs. optimization problems; polynomial time; certificates and polynomial-time verification; the classes P, NP, NP-hard, and NP-complete; many-one (Karp) reductions and reduction direction; Cook-Levin and SAT; the standard NP-complete family (3-SAT, CLIQUE, VERTEX-COVER, INDEPENDENT-SET, HAMILTONIAN-CYCLE, SUBSET-SUM); what P vs. NP is and is not.
- **Applied skills**: convert an optimization problem to its decision version; verify a proposed certificate in polynomial time; outline a reduction and argue correctness in both directions (yes-maps-to-yes and no-maps-to-no); distinguish NP-hard from NP-complete; explain to a non-theorist why "NP-complete" does not mean "unsolvable in practice."
- **Common traps**: reducing in the wrong direction (to show B is hard you reduce a known-hard A *to* B, not B to A); claiming NP means "non-polynomial" (it means nondeterministic-polynomial / polynomially verifiable); assuming NP-complete instances are always slow (many real instances are easy; the hardness is worst-case); confusing NP-hard (at least as hard, need not be in NP) with NP-complete (in NP and NP-hard); thinking a fast heuristic that "usually works" disproves NP-completeness.

## Chapter 8: Randomized, Approximation, and Practical Algorithm Engineering
- **Core questions**: When is a coin flip a legitimate part of an algorithm, and what guarantee do you get in return? If you cannot solve a hard problem exactly, how good a solution can you provably promise? Before you optimize, how do you find out what is actually slow?
- **Key concepts**: randomized quicksort and expected analysis; Las Vegas vs. Monte Carlo algorithms (always-correct/random-time vs. fixed-time/maybe-wrong); the difference between expectation and high probability; the greedy ln n approximation for set cover; the maximal-matching 2-approximation for vertex cover; approximation ratios and what they bound; benchmarking and profiling discipline; responsible use of library implementations.
- **Applied skills**: compute the expected number of comparisons or restarts for a randomized choice; classify an algorithm as Las Vegas or Monte Carlo and state its guarantee; prove the vertex-cover 2-approximation via the disjoint edges of a maximal matching; read a profile and identify the real hotspot before changing code.
- **Common traps**: conflating expected runtime with worst-case runtime (randomized quicksort is expected O(n log n) but worst-case O(n^2)); confusing Las Vegas (correct, random time) with Monte Carlo (fixed time, possibly wrong); reporting an approximation algorithm's output as optimal; assuming a 2-approximation always returns a solution twice as large (it never exceeds twice optimum, often much better); optimizing the part of the code that "feels" slow instead of the measured hotspot; micro-optimizing constants while leaving an asymptotically worse algorithm in place.

## Capstone
Students complete an algorithmic investigation that begins with a real problem and ends with a defensible implementation and analysis. The project must (1) state the computational problem precisely, including input format, size, and what counts as feasible and optimal; (2) model the input as a concrete structure (array, graph, DP state space); (3) compare at least two algorithmic strategies, justifying why each fits or fails; (4) prove correctness for the chosen exact method, or state and justify an approximation or randomized guarantee for a hard problem; (5) implement and test the solution, including adversarial and edge-case inputs; (6) analyze time and space complexity with a stated model; (7) measure performance on meaningful input sizes and compare the measured curve to the predicted asymptotics; and (8) present the tradeoffs in writing for a technical reader. Strong project candidates: route planning on real map data, course or exam scheduling, document-similarity via edit distance or LCS, resource allocation, network design with MST, an approximate set-cover deployment problem, sequence alignment in bioinformatics, or solving and analyzing a constraint puzzle (Sudoku, n-queens) with backtracking plus pruning.

## Assessment Ideas
- Weekly problem sets combining proofs, recurrence solutions, asymptotic bounds, and small implementation tasks.
- In-class proof checks on loop invariants, induction, exchange arguments, cut properties, and reduction direction.
- Programming labs graded on correctness, edge cases, asymptotic behavior, code clarity, and empirical timing evidence.
- Short quizzes on algorithm selection, graph traces, DP state design, and complexity classification.
- Capstone rubric covering problem formulation, algorithm choice, proof quality, implementation reliability, test design, performance analysis, and communication.
